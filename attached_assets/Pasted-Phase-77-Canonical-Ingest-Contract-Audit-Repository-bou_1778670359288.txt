  Goal:
  Create a clear canonical ingest contract for how TikTok/video concepts move from Hagen analysis into hagen-ui, concept
  library, customer assignment, and customer-specific overrides.

  This phase is primarily audit + contract documentation. Do not perform a broad refactor yet. Only make tiny code/docs
  edits if needed to clarify current behavior.

  Business rules to encode:
  - Upload-confirm is generally the preferred CM approval UX.
  - /analyze-rate-v1 in old Hagen seemed better at objective preselected metadata such as number of people, filming
  time, difficulty. Compare that logic with current upload-confirm/analyze flow.
  - A successful Gemini/API response is not the same as successful metadata extraction.
  - Objective-ish fields are safer:
    - number of people
    - number of scenes
    - script length / script mode
    - difficulty / replicability
    - visible setup complexity
    - business types, and this must support multiple business types
  - Budget should not be part of the canonical model. It is not useful.
  - Production tips, title, why it works, humor analysis, and similar fields are subjective and must be treated as CM-
  editable/candidate text, not trusted facts.
  - Script/manus is optional:
    - some TikTok concepts have only overlay text
    - some are 3-second sketches
    - some have short dialogue
    - some have longer scripts
  - Save means approval. When a CM saves an ingested concept, it should become active in the library immediately.
  - Library edits should modify reusable core concept data.
  - Customer-specific adaptations should live outside the core library concept, in customer assignment / customer
  concept override structures.
  - AI quality/recommendation score should not be treated as reliable now.
  - Current niche focus is hospitality: restaurant, bar, cafe, hotel/venue-style businesses.
  - Trend level is not an AI-inferred truth now. If used later, it should be CM-marked/manual.
